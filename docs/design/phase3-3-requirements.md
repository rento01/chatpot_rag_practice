## 背景

Phase 3-2 でベクトル検索を実装し、意味的に近いチャンクをヒットさせられるようになった。しかし：

- **BM25 の強み**: 固有名詞・専門用語・型番など、キーワードが一致しないと意味的に遠い場合でも正確にヒットする
- **ベクトル検索の強み**: キーワードが異なっても意味が近ければヒットする
- **互いの弱点**: BM25 は言い回しの違いに弱く、ベクトル検索はキーワードの完全一致に弱い

両者を統合することで弱点を補い、より安定した検索精度を実現する。

Phase 3-2 では BM25 コードをコメントアウトで残存させており、Phase 3-3 で再利用する前提で設計されている。

---

## 目的

- BM25 とベクトル検索の結果を RRF（Reciprocal Rank Fusion）で統合し、安定したヒットを実現する
- 「キーワード一致か意味的な近さか」に依存しない検索を体験する（学習ポイント）
- 検索パラメータを `.env` で調整可能にして、効果の違いを観察しやすい環境を整える

---

## 要件

| ID | 要件 |
|---|---|
| R-01 | BM25 検索とベクトル検索の両方を実行し、それぞれの結果リストを取得できること |
| R-02 | RRF でスコアを統合し、1つの `list[SearchResult]` として返せること |
| R-03 | BM25 のみヒット・ベクトルのみヒット・両方ヒット・どちらもヒットなし の全ケースで正しく動くこと |
| R-04 | ハイブリッド検索に関わるパラメータ（候補数・RRF の k 定数）が `.env` から設定できること |
| R-05 | `n_results` がコレクションのチャンク数を超えないように制御すること（RI-05 対応） |

---

## RRF の仕組み

```
BM25 で上位 bm25_top_k 件を取得（スコア > 0 のもの）
  +
ベクトル検索で上位 vector_top_k 件を取得

→ 各チャンクの RRF スコアを計算する
   score = Σ ( 1 / (k + rank_i) )  ※ 両リストに登場するなら加算
   k のデフォルト: 60（RRF の標準定数）

→ RRF スコアで降順ソートして上位 top_k 件を返す
```

`SearchResult.score` には RRF スコアを格納する。

## 懸念事項

| ID | 内容 | 対応方針 |
|---|---|---|
| C-01 | BM25 は全チャンクを取得（`col.get()`）してからスコア計算するため、チャンク数が増えると遅くなる | 教材規模では問題なし。Phase 6 以降で改善を検討 |
| C-02 | ベクトル embedding が `None` のチャンクが混在する場合、ベクトル検索が空リストを返す | BM25 のみの結果で RRF を実行する設計とする |
| C-03 | BM25 の候補数（`bm25_top_k`）とベクトル検索の候補数（`vector_top_k`）の設定粒度 | まずは `bm25_top_k` と `search_top_k` を兼用し、段階的に分離する設計とする |

## Bolt 設計への引き継ぎ

- `ChromaVectorDB.search` 内の BM25 コメントアウトを解除して再利用する
- RRF の実装は `search` メソッド内にインラインで実装する（別クラス・別関数への切り出しはしない）
- `SearchResult.score` には RRF スコアを格納する
- `VectorDB.search` のシグネチャは変更しない
- パラメータは `settings.py` 経由で読み込み、`.env` から切り替えられるようにする
