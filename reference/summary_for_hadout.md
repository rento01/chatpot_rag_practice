# 教材資料反映用サマリ (summary_for_hadout)

このファイルは、`rag-chat-template` リポジトリの実装内容を **既存の handout 資料
（xlsx ロードマップ / pptx 解説 / pptx 参考知識）に反映するための中間整理** です。

- 既存資料の意図と方針は維持し、**Phase 設計・作業内容・時間配分・学習密度・AI 駆動開発観点** を具体化することがゴール
- そのまま「ロードマップ表」「スライド見出し」「スライド本文」として転記しやすい粒度に揃えてある
- 内訳は実 Issue / ROAD_MAP / SETUP_GUIDE / 実装ファイルから抽出済み（出典は §8 参照）

## 1. リポジトリ全体の概要

### 1.1 ねらい
- **初めて RAG (Retrieval Augmented Generation) に触れる社員** が、ローカル環境で
  「動くチャット → 文書取り込み → 検索 → 観測 → 品質改善 → AWS 移行 → 評価」までを
  段階的に体験して学ぶための教材テンプレ
- AI 駆動開発（Claude Code を中心に、bolt 単位で Issue → PR → AI レビュー → マージ）の
  サイクル自体も学べる構成

### 1.2 構成・技術要素

| レイヤ | 採用技術 | 教材としてのねらい |
| --- | --- | --- |
| Backend | Python 3.11+ / FastAPI / SQLAlchemy / alembic | RAG の中身を読み解きやすい最小実装 |
| Frontend | Next.js (App Router) / React 19 | チャット + ファイル取り込み画面の最小UI |
| RDB | PostgreSQL | 会話履歴・文書メタの保管 |
| Vector DB | ChromaDB (ローカル) / OpenSearch (AWS, 雛形) | プロバイダ切替で構造を学ぶ |
| LLM | Ollama (ローカル) / AWS Bedrock (AWS, 雛形) | 同上 |
| 観測 | Langfuse (任意, 初期は no-op) | Phase 4 で導入 |
| IaC / CI | Terraform / GitHub Actions (OIDC 想定) | Phase 7 |
| 開発 | Claude Code + Codex + bolt-planning skill | AI 駆動開発の運用練習 |

### 1.3 学習者が最終的に得るもの

- RAG パイプラインの **構成要素 (取り込み / チャンク / 埋め込み / 検索 / 観測 / rerank / 評価)** が
  自分の言葉で説明できる
- 1 つずつ実装し、**結果を観測して品質改善する** サイクルを回せる
- 開発の進め方として **bolt 単位の Issue → AI 実装 → AI レビュー → 人間判断 → マージ** を体感する
- ローカル → AWS 移行の **インフラ実装 (Terraform + OIDC + Bedrock + OpenSearch)** の感覚を掴む

### 2.2 依存関係グラフ（pptx ロードマップ図用）

```
G0 環境構築
 └→ G1 基本 RAG
      └→ G2 ベクトル + ハイブリッド
           ├→ G3 観測 (任意で並行)
           └→ G4 検索品質改善
```

## 3. Phase 別詳細（pptx スライド本文用）

各 Phase は **スライド 1 枚に収まる粒度** で書いてある。
詳細を深掘りしたい場合は対応 Issue / ROAD_MAP の節を参照。

### 3.1 G0 — 環境と構成理解 (Phase 0-1 / 0-2 / 1)

- **目的**: 自分の PC でテンプレが動く状態を作り、コード全体の「読み所」を頭に入れる。
- **作業内容**:
  - Docker Desktop / Ollama インストール、テンプレ clone + `{Name}-rag` 初期化、`docker compose up -d`、`ollama pull`、ブラウザで動作確認
  - `backend/main.py` → `dataModels.py` / `schemas.py` → `rag.py` (薄ラッパ) → `llm/` / `vector_db/` → `frontend/src/app/page.tsx` の順で読む
  - `.env` で別モデルを試し、ログを眺める癖をつける
- **学習スコープ**: セットアップ手順の理解、Docker / Ollama / FastAPI / Next.js の全体像把握
- **前提知識**: Docker の基本、Python・TypeScript の読解レベル
- **成果物**: 動くローカル環境、構成理解メモ（任意）
- **時間係数**: **2**
- **AI 委任度**: **高**（手順は SETUP_GUIDE.md にあるため AI に「`SETUP_GUIDE.md` に沿って立ち上げて」と任せやすい）
- **つまずきポイント**: Linux ホスト Ollama の `OLLAMA_HOST=0.0.0.0` 設定 / モデル未 pull で 404
- **次への接続**: 「`NotImplementedError` を埋めればこう動く」が頭にあれば G1 に進む

### 3.2 G1 — 基本 RAG (Phase 2-1 / 2-2)

- **目的**: PDF を取り込んで、キーワードで関連箇所を引っ張ってくる「最初の RAG」を動かす。
- **作業内容**:
  - `backend/rag.py` の `index_document` 完成（テキスト抽出 → 分割 → upsert）
  - `backend/vector_db/chroma.py` の `upsert` / `search`（BM25 ベース）を実装
- **学習スコープ**: チャンク設計、ChromaDB の collection / metadata、BM25、日本語トークナイズ
- **前提知識**: 簡単な Python / SQL / 検索の概念
- **成果物**: `/ingest` で status=`ready` になる + RAG モードでヒットが返る
- **時間係数**: **3**
- **AI 委任度**: **中**（チャンク粒度や日本語トークナイザの選定は人間判断 / 実装は AI に書かせやすい）
- **つまずきポイント**: BackgroundTasks の挙動、日本語向け bi-gram トークナイザの必要性
- **次への接続**: 「キーワードで拾えないクエリ」を実感するのが G2 への動機

### 3.3 G2 — ベクトル化とハイブリッド検索 (Phase 3-1 / 3-2 / 3-3)

- **目的**: 言い回しを変えた質問にも答えられるようにし、ランクの統合方法を体得する。
- **作業内容**:
  - 取り込み時に embedding を付与（Phase 3-1）
  - ベクトル検索を実装し、`.env` でパラメータを変えられるように（Phase 3-2）
  - キーワード + ベクトルを RRF 等でマージ（Phase 3-3）
- **学習スコープ**: 意味空間、k-NN、入力長制約、RRF、検索パラメータの観察
- **前提知識**: 線形代数の基本（コサイン類似のイメージ）、ベクトル DB の概念
- **成果物**: 同じ質問でも検索品質が体感で上がっていること
- **時間係数**: **5**
- **AI 委任度**: **中**（パラメータ調整の判断は人間 / 実装と検証スクリプトは AI）
- **つまずきポイント**: 埋め込みモデルの入力上限、ハイブリッド時の重み付け
- **次への接続**: 「何が起きているか」が知りたくなったら G3 へ

### 3.4 G3 — 観測導入 (Phase 4)

- **目的**: チャットと取り込みの内部処理を span ツリーで可視化し、改善サイクルを回せるようにする。
- **作業内容**:
  - Langfuse のキーを `.env` に設定
  - `backend/tracing.py` 既存ラッパを有効化、`backend/main.py` / `rag.py` に span / generation を入れる
- **学習スコープ**: 観測粒度（retrieval / generation / save）、no-op 切替
- **前提知識**: ログとトレースの違い、SaaS への API キー設定
- **成果物**: Langfuse ダッシュボードで 1 リクエストが span ツリーで見える
- **時間係数**: **1**
- **AI 委任度**: **高**（API 設計通りに span を入れる作業）
- **つまずきポイント**: 鍵未設定で完全に動くこと（no-op）を確かめずに「動かない」と勘違いする
- **次への接続**: 観測しながら G4 の改善が回ると効果が見える

### 3.5 G4 — 検索品質改善 (Phase 5 / 6)

- **目的**: 検索のヒットしやすさと回答粒度のずれを吸収し、本当に近いものだけを LLM に渡す。
- **作業内容**:
  - 親子チャンク導入（検索は子、コンテキストは親）
  - rerank を `backend/rag.py` に追加（まず heuristic、必要なら cross-encoder）
- **学習スコープ**: 検索粒度 / 回答粒度の分離、メタデータ設計、rerank の効きどころとコスト
- **前提知識**: G2 までで触れた検索の挙動
- **成果物**: rerank 有無で品質を比較できる + 親重複排除されたコンテキスト
- **時間係数**: **5**
- **AI 委任度**: **中**（実装は AI / 効果検証の判断は人間）
- **つまずきポイント**: 親子の id 設計、rerank の段の置き場所
- **次への接続**: 改善が地に足ついたら G5 の AWS 移行へ

### 3.6 G5 — AWS 本番化 (Phase 7)

- **目的**: ローカルテンプレを AWS 上で動かす。CI からインフラ更新とアプリデプロイができる状態にする。
- **作業内容**:
  - `terraform/main.tf` のサブネット / IGW / NAT / SG / Bedrock VPCe / RDS / OpenSearch / ECR / ECS / ALB を実装
  - `backend/llm/bedrock.py` / `backend/vector_db/opensearch.py` の本実装
  - `.github/workflows/` に OIDC で `terraform apply` + ECR push + ECS deploy
- **学習スコープ**: PrivateLink、OpenSearch Serverless の AccessPolicy / NetworkPolicy、OIDC
- **前提知識**: AWS の基本サービス、Terraform、GitHub Actions
- **成果物**: AWS 上の ALB 経由でチャットが返る + CI から apply / deploy できる
- **時間係数**: **8**（最重）
- **AI 委任度**: **低**（IAM / セキュリティ判断・課金/権限の人間確認が多い）
- **つまずきポイント**: VPC ルーティング、AccessPolicy の構文、OIDC ロール設定
- **次への接続**: 本番が動いたら G6 で品質を数値化

### 3.7 G6 — 品質評価 (Phase 8)

- **目的**: RAG の検索 / 生成品質を主観ではなく数値で測り、改善前後を比較する。
- **作業内容**:
  - `eval/` 新設、評価データセット（質問 + 期待回答 + 期待ソース）の整備
  - `ragas` で faithfulness / answer_relevancy / context_precision
  - 結果を `docs/eval/` に Markdown ログとして残す
- **学習スコープ**: 評価データの作り方、主観品質と指標のずれ、Langfuse トレースとの突合
- **前提知識**: G3 で取った Langfuse の trace、評価の基本概念
- **成果物**: 1 コマンドで評価が回り、レポートが出る
- **時間係数**: **3**
- **AI 委任度**: **中**（評価セットの設計は人間 / スクリプト整備は AI）
- **つまずきポイント**: 評価セットの代表性、指標値の解釈
- **次への接続**: 評価結果を入力に、G4 を再度回すループ

## 5. 実装の意図と手順（pptx ロードマップ解説スライド用）

### 5.1 なぜこの Phase 構成か

- **「動くもの → 簡単な検索 → 深い検索 → 観測 → 品質改善 → 本番化 → 評価」** は、RAG プロジェクトの実プロセスとほぼ同じ順
- **NotImplementedError による穴埋め式**: 取り込み API / Vector DB ラッパ / プロバイダ切替の枠は既に揃っており、学習者は「中身」を埋めるだけ。設計判断と実装判断を分けやすい
- **観測 (G3) を品質改善 (G4) より前に置く**: 改善の効果を測る基盤を先に整える方が、改善が定量化できる
- **AWS 移行 (G5) を改善 (G4) の後に置く**: ローカルで品質に納得してから本番化することで、本番デバッグの泥沼を避ける
- **評価 (G6) を最後に置く**: 全パイプラインが揃ってから RAGAS を回すと、評価が回す全プロセスを横断する

### 5.2 各 Phase で得られる学習成果

- G0: アプリの読み所と「自分のリポジトリで動かす」起点
- G1: チャンク・キーワード検索という RAG の最小核
- G2: ベクトルの意味空間とランク統合の感覚
- G3: 改善サイクルを支える「観測」の重要性
- G4: 検索粒度と回答粒度の分離 + rerank の効きどころ
- G5: 構成を「ローカル前提のラッパ」から「本番マネージドサービス」に移す手順
- G6: 「主観品質」と「指標値」のずれを自分の頭で解釈する経験

### 5.3 つまずきやすい点 (FAQ スライド用)

- **モデル未 pull で 404 が返る** → §SETUP_GUIDE §3.2 を参照
- **Linux ホスト Ollama に届かない** → SETUP_GUIDE §3.1 の `OLLAMA_HOST=0.0.0.0:11434`
- **BM25 が日本語で効かない** → 日本語向け前処理（bi-gram など）を入れる
- **Langfuse でリクエストが見えない** → `.env` のキー設定と `tracing.py` の有効化
- **AWS で接続できない** → SG / Subnet / IAM の 3 点セットを順に確認

## 6. AI 駆動開発を意識した設計（章末総論）

### 6.1 役割分担の基本方針

| 種別 | 人間 (学習者) の主担当 | AI (Claude Code 等) に任せやすい |
| --- | --- | --- |
| 設計判断 | Phase 順序、bolt 分割の単位、検索パラメータの当たり方の解釈 | 既存実装の差分把握、Issue 本文→bolt 設計の起こし |
| 実装 | アーキ判断を伴う実装 (例: G5 の IAM / VPC) | 雛形からの実装 (G1 / G3 / G6 の多く) |
| レビュー | 設計判断とセキュリティ部の最終決定 | 差分の自己レビュー / Codex による独立レビュー |
| ドキュメント | Phase 設計判断の言語化 | 文章の要約・整形・参照リンク調整 |

### 6.2 bolt 単位の指示の出し方

このリポジトリでは **bolt-planning skill** を採用しており、Issue を 1 つの bolt =
1 つの PR として扱う。各 bolt は次の節立てで設計してから着手する。

- 目的 / 作るもの / 作らないもの / 対象ファイル / 実装方針 / ドキュメント更新 / 完了条件 / 確認事項 / 次の bolt への引き継ぎ

学習者には「Issue 本文 → bolt 設計 → 実装 → AI レビュー → 人間判断 → マージ」の 1 サイクルを
**G1 (Phase 2-1) で必ず通させる**。サイクルの感覚が掴めるとそれ以降の Phase は自走しやすい。

### 6.3 AI レビューの位置づけ

このテンプレは **Codex レビュー** を `.claude/codex-mode` で `review-merge` 設定にしている。
PR 作成後、自動でレビューが走り、`STATUS=clean` ならマージ判断、`findings` なら修正が必要。

- **必ず人間確認で停止する条件**: スキーマ変更、セキュリティ判断、設計トレードオフ、要件曖昧
- 教材として、各 Phase で **最低 1 回は Codex 指摘を受けて修正する経験** をさせると効果的

### 6.4 学習者にどう説明するか（スライド見出し例）

- 「AI に書かせる前に何を決めるか」（人間判断の整理）
- 「bolt 設計の節立てが全部埋まるか」（埋まらないと指示が曖昧）
- 「Codex の指摘を受け入れる/受け入れない判断」（合意済設計を変えない自由）
