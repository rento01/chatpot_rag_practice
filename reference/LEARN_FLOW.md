# RAG 学習企画の進め方 (LEARN_FLOW)

このドキュメントは、RAG 学習企画の進め方を学習者・運営メンバーで共有するためのものです。

このプロジェクトでは、単に動くものを作るだけでなく、

- 各 Phase の Issue を起点に、設計・実装・PR・レビュー・進捗管理を回す
- 実装だけでなく、**学習記録 / レビュー記録** も残す

ことを目的としています。

「次に何をすればよいか分からない」状態にならないよう、初回の人もこの順番で読み進めれば作業を始められる構成にしています。

詳細な実装ロードマップは [`ROAD_MAP.md`](ROAD_MAP.md) と [`ISSUES.md`](ISSUES.md)、環境構築は [`SETUP_GUIDE.md`](SETUP_GUIDE.md) を参照してください。

---

## 1. 大まかな作業プロセス

```text
Phase Issue を確認
   ↓
設計 (必要なら子 Issue を作成)
   ↓
bolt を切って実装 + taskLog を残す
   ↓
feature ブランチ → PR
   ↓
自分で確認 → マージ
   ↓
週次 MTG でメンバーレビュー
   ↓
GitHub Projects で進捗管理
```

### 1.1 Phase ごとの Issue を確認する

- 各 Phase ごとに用意された Issue を確認します（一覧は [`ISSUES.md`](ISSUES.md) / GitHub Issues）。
- まずは、その Issue を達成するための **設計** から入ります。
- 詳細設計書を作るかどうか、どの程度細かく設計するかは作業者の判断に任せます。手で図を書く、Markdown でメモを残す、Claude と対話して整理する、どれでも構いません。
- 必要に応じて、**子 Issue を自由に作成して構いません**。1 つの Phase Issue が大きすぎると感じたら、分割した方が PR も小さく保てます。
- 子 Issue を作成する場合は、GitHub の Issue ページ右側の `Relationships` から **parent** を指定すると、親 Issue 側にも子 Issue が表示されて見通しが良くなります。

### 1.2 bolt を切って実装する

- 設計が固まったら、**bolt** という小さな実装単位に切って着手します。
- bolt の命名・出力フォーマット・判断基準は [`.claude/skills/bolt-planning/SKILL.md`](../.claude/skills/bolt-planning/SKILL.md) にまとめてあります。任意で使って構いません。
- bolt の粒度は、**作業の見通しが立ち、PR としても確認しやすい単位** にします。1 bolt = 1 PR を原則として、迷ったら小さい方に倒してください。
- 実装時には **taskLog** を残します。
  - 置き場所: `docs/taskLog/`
  - 含める項目（最低限）:
    - 目的
    - 変更内容
    - 変更ファイル
    - 判断理由
    - 実行した確認
    - 残課題
- taskLog を残す目的は、後から自分や他のメンバーが「何を考えてこの実装にしたか」を追えるようにするためです。コミットメッセージや PR 本文と重複しても構いません。

## 2. PR 作成とレビューのフロー

実装は、なるべく Git の **feature ブランチを切り、PR を出す** 流れで進めます。レビューの有無に関わらず、変更履歴を残すことで後から振り返れるようにするためです。

### 2.1 feature ブランチを作成する

- ブランチ名は、後から見て **Issue や Phase との関係が分かるもの** にします。
- 例:
  - `feature/<issue-number>-short-name`
  - `fix/<issue-number>-short-name`
  - `docs/<issue-number>-short-name`
  - `infra/<issue-number>-short-name`

### 2.2 PR を作成する

- 実装が一区切りしたら PR を作成します。
- **Claude に PR を作らせる運用も推奨** します。PR タイトル・本文・Summary・Test plan までを Claude に下書きさせ、それを自分でレビューすることで、「人が書いた前提のレビュー練習」にもなります。
- PR 本文には、bolt 設計（目的 / 作るもの / 作らないもの / 完了条件 / 引き継ぎ）を冒頭に貼ると、レビュー時の文脈が伝わりやすくなります。

### 2.3 自分で確認してマージする

- レビュー待ちで作業が止まらないよう、**自分で確認して問題なければマージして進めて構いません**。
- ただしマージ前に、最低限以下は自分で確認してください。
  - 差分（`git diff` / GitHub の Files changed）
  - 動作（ローカルで動かす / 該当機能を触ってみる）
  - 意図（Issue / 設計と実装がズレていないか）
- マージしたあとも、その PR は MTG でレビュー対象になります。**close したから終わり、ではない** 点に注意してください。

### 2.4 週次 MTG で全員レビューする

- PR は週 1 回の MTG で、メンバー全員で確認し、レビューし合います。
- open 中の PR だけでなく、**close した PR も対象** にします。
- close 済み PR についても、設計意図・実装内容・レビュー観点を MTG 内で説明できるようにしておくと、学習効果が高くなります。

## 3. Issue と PR の進捗管理

Issue と PR は、以下の GitHub Projects で管理します。

- https://github.com/orgs/PlusValueLab/projects/1

運用方針:

- **自分の view を作る**: `new view` で自分の名前の view を作成し、自分の作業が一覧できるようにしておきます。
- **Issue と PR を定期的に追加する**: `add new item` で、自分が着手する Issue / 作成した PR をまとめて追加してください。少なくとも週に 1 回は棚卸しする想定です。
- **自動追加は将来的に検討**: 現状は手動運用ですが、ワークフロー連携で自動追加するよう拡張する可能性があります。
- **`[NAME-rag]` で filter**: タイトル先頭に `[NAME-rag]` を付けておくと、その文字列で filter をかけて自分の作業だけ抽出できます。
- **close 済み PR の管理**: close した PR も、**MTG レビュー前 / レビュー後** で分けて管理しておくと、抜け漏れ防止になります。
