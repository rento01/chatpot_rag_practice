## Issue: Phase 2-1 — ファイル取り込みを動かす

### 目的
PDF をコレクションに取り込めるようにし、後段の検索 / RAG が動く土台を作る。

### 方針
取り込み API は完成済みなので、その先で動く **テキスト抽出 → 分割 → ベクトル DB への保存** のロジックを埋める。`/ingest` で PDF を送ったときに status が `error` で止まる箇所をなくし、`ready` まで完走させるのがこの Phase のゴール。
参考になる場所は `backend/rag.py`（取り込みロジック本体）、`backend/vector_db/chroma.py`（Chroma 側の保存処理）、`backend/main.py` の取り込み API の入口あたり。

### 学習ポイント
- 文書をどの単位でチャンクに切るか、その切り方が後段の検索体験にどう響くか
  * Phase2では固定長(500文字など)でOK。
- OCRでテキスト取り込みをした際の、適切なデータクレンジングを作れるか
- バックグラウンドタスクで重い処理を回すときの注意点（非同期・パイプ処理）

### 完了判断
- [ ] `/ingest` ページから PDF をアップロードして status が `ready` になる
- [ ] アップロード後、Chroma 側のコレクションにチャンクが入っていることを確認できる
- [ ] 数十ページ規模の PDF でも取り込み API がタイムアウトせず非同期に処理が完了する

### 参考
- ChromaDBのGUI確認ツール: https://github.com/BlackyDrum/chromadb-ui
> 色々種類はあるのでお好みで。

## Issue: Phase 2-2 — キーワード検索を実装する

### 目的
RAG モード ON のチャットで、コレクション内の文書から関連箇所を引っ張ってきて回答に使う「最初の検索」を動かす。

### 方針
キーワード一致ベースの検索（BM25 など）を `backend/vector_db/chroma.py` の検索処理に追加し、`backend/rag.py` 側のコンテキスト組み立てで使われるようにする。
日本語は空白で単語が切れないため、トークナイズの粒度が結果に強く効く。素朴な実装で一度動かしてから、ヒットしないクエリで何が起きているかを観察するのがおすすめ。

### 学習ポイント
- BM25 の挙動と、日本語ドキュメントでの効きどころ
- 日本語向けのトークナイズ（単語分割）の実装方法
- ヒット 0 のときに「資料に記載がありません」と返すための分岐点

### 完了判断
- [ ] 取り込み済みコレクションを選んで RAG モード ON で質問するとヒットしたチャンクが回答の根拠として返る
- [ ] ヒットがない場合は「資料に記載がありません」と返る

## Issue: Phase 3-1 — embedding を生成する

### 目的
Phase 2-1 で取り込んだチャンクに、後段のベクトル検索で使う埋め込みベクトルを付ける。

### 方針
`backend/llm/` に用意してある埋め込みモデルのラッパを使い、取り込み時にチャンクごとの embedding を生成して `backend/vector_db/chroma.py` 側に保存する。
長いチャンクはモデルの入力上限に当たることがあるので、入力長やバッチサイズの扱いを意識する。取り込みは時間がかかるので、進捗が見える工夫もしておくと体験が良くなる。

### 学習ポイント
- ベクトル検索と意味空間への大まかな理解
- 埋め込みモデルの選定とベクトルの次元設定
- 埋め込みモデルの入力長制約と、チャンクサイズとの相性
- 取り込み中の進捗ログ設計（ユーザー体験との関係）

### 完了判断
- [ ] 取り込み完了後、Chroma 側でチャンクごとに embedding が保存されている
- [ ] 取り込み中の進捗が backend ログで追える

## Issue: Phase 4 — Langfuse でトレースを取る

### 目的
チャットと取り込みの内部処理を可視化し、品質改善のサイクルを回せるようにする。

### 方針
すでに薄いラッパだけ用意してある `backend/tracing.py` を有効化し、`backend/main.py` のチャット処理と取り込み処理に「検索 / 生成 / 保存」など節目ごとの記録を入れていく。Langfuse の鍵を未設定でもアプリは動く（no-op）状態は維持する。

### 学習ポイント
- どの単位で観測すれば改善の手がかりが見えるか
- 観測の追加コスト（コード量・遅延）と得られる情報量のバランス

### 完了判断
- [ ] Langfuse のダッシュボードで 1 件のチャットを span ツリーとして開ける
- [ ] `LANGFUSE_*` を空にしてもアプリは従来通り動く

### 参考
- langfuse repo: https://github.com/langfuse/langfuse
> ローカルでもwebでも可。ローカルの方が扱いが素直

## Issue: Phase 5 — 親子チャンクで分割を改善する

### 目的
検索のヒットしやすさと、LLM に渡すコンテキストの読みやすさを両立させる。

### 方針
ドキュメントを **親（大きめ）→ 子（小さめ）** の階層に分割し、`backend/rag.py` と `backend/vector_db/chroma.py` を改修する。
検索は子チャンク単位で当て、LLM に渡すコンテキストは親チャンク単位で渡す、という粒度の分離を目指す。重複した親の取り扱いも忘れずに。

### 学習ポイント
- 構造化文書をデータ前処理への理解
- 検索粒度と回答粒度のずれをどう吸収するか
- チャンクのサイズをどの粒度で切り分けるか
- ベクトル化する文書と、推論用文書をmeta dataでどう制御するか

### 完了判断
- [ ] 同じ質問でも、Phase 3-3 と比べて広い文脈が回答に使われる
- [ ] 推論LLMに渡す子チャンクの文脈が失われていない
- [ ] 同じ親チャンクの子が複数ヒットしても、コンテキストとしては親単位で重複が排除される

## Issue: Phase 6 — rerank を実装する

### 目的
検索結果のうち、本当に質問に近いものだけを LLM に渡して回答品質を上げる。

### 方針
ハイブリッド検索 + 親子チャンクの結果に対して、`backend/rag.py` で rerank 処理を入れ、`backend/main.py` のチャット処理に組み込む。
最初は軽量な heuristic（クエリ語の一致度合いなど）から始めるのがおすすめ。必要に応じて cross-encoder のような重めの手段に置き換えていけばよい。

### 学習ポイント
- rerank のコストと、得られる品質向上のバランス
- rerank が効かないクエリの傾向
- 親チャンクの統合をどの段で行うことが最適か判断する

### 完了判断
- [ ] 検索 → rerank → 上位 N 件、というフローがチャット処理に入っている
- [ ] rerank の有無で回答品質を切り替えて比べられる
- [ ] Phase5 よりも回答速度・品質が向上する

## Issue: Phase 7 — AWS 移行 (Terraform + GitHub OIDC)

### 目的
ローカルで動かしてきたテンプレを、AWS 上で動かす。

### 方針
`terraform/main.tf` でネットワーク・データベース・ベクトル DB・実行基盤を整え、LLM とベクトル DB を AWS マネージドサービス（Bedrock / OpenSearch）へ切り替える。CI からのデプロイ（OIDC）も整える。
作業範囲が広いので、`.claude/skills/bolt-planning/SKILL.md` を使って **terraform / Bedrock 切替 / OpenSearch 切替 / CI** あたりを別 bolt に切り分けて進めるのが安全。参考になる場所は `terraform/main.tf` / `backend/llm/bedrock.py` / `backend/vector_db/opensearch.py` / `.github/workflows/`。

### 学習ポイント
- PrivateLink 経由でマネージド LLM を呼ぶ構成
- OpenSearch Serverless の AccessPolicy / NetworkPolicy
- GitHub Actions から OIDC でインフラ apply / アプリ deploy を流す段取り

### 完了判断
- [ ] AWS 上で ALB 経由のチャットが返ってくる
- [ ] `.env` で LLM プロバイダ / Vector DB プロバイダを Bedrock / OpenSearch に切り替えられる
- [ ] GitHub Actions から OIDC でインフラ更新とアプリのデプロイができる

## Issue: Phase 8 — RAGAS で品質評価を回す

### 目的
RAG の検索 / 生成品質を、主観ではなく数値で測れるようにする。

### 方針
評価用のデータセット（質問 + 期待回答 + 期待ソース）を整え、評価ライブラリ（`ragas` など）を使って faithfulness / answer_relevancy / context_precision のような指標を取る。結果はリポジトリ内のドキュメント（例: `docs/eval/` に Markdown）として残し、改修の前後で比較できる形にする。
新規に作る可能性が高いディレクトリは `eval/`（スクリプトとデータ）と `docs/eval/`（結果ログ）。

### 学習ポイント
- 評価データセットの作り方（典型 / 例外 / 失敗例）
- 主観品質と評価指標のずれをどう解釈するか
- Phase 4 で取った Langfuse のトレースと突き合わせる読み方

### 完了判断
- [ ] 評価が 1 コマンドで回り、レポートが出力される
- [ ] Langfuse のトレースと RAGAS のスコアを突き合わせて品質を分析できる
