# bolt-planning

このリポジトリでは、Claude Code が大きな Issue を一気に実装するのではなく、
**bolt** と呼ぶ小さな作業単位に分けてから実装を進める。
この skill は、各 bolt の **「何を作るか」を整理する** ために使う。

> tasklog や architect の運用そのものは、この skill のスコープ外。
> 必要になったら別 skill / 別 Issue で扱う。

---

## 1. 目的

この skill は、以下のために使う。

- 大きな Issue を bolt 単位に分割する
- 各 bolt で **作るもの / 作らないもの** を明確にする
- 実装前に **作業範囲を整理** することで、作りすぎ・脱線・過剰実装を防ぐ
- 次の bolt に **引き継ぎやすい形** で成果物を定義する
- 設計判断や曖昧な前提を、実装前にユーザーへ確認する

---

## 2. いつ使うか

以下のタイミングで、この skill を発動する。

- 新しい Issue に着手する直前
- 1 つの Issue が大きく、複数の責務をまとめて持っているとき
- reference/ROAD_MAP.md の Phase を実装する直前
- 既存 bolt の途中で「思ったより大きい」と気づいたとき
  （bolt を分割し直してから続ける）

逆に、以下では使わなくてよい。

- typo 修正・コメント修正など、明らかに 1 ステップで終わるもの
- 既に bolt 設計済みで、その続きを実装するとき
  （既存 bolt の「次への引き継ぎ」をそのまま使う）

---

## 2.5 bolt分割判定

Issue または Phase に着手した際は、

まず bolt 分割が必要かを判定する。

分割不要の場合は bolt-0 のみで進めてよい。

判定理由を説明すること。

## 3. bolt の命名規則

bolt 名は、必ず以下の形式にする。

```
bolt-0
bolt-1
bolt-2
bolt-3
```

- 数字は **0 から開始** する
- `bolt-zero` や `bolt_0` などの **別表記は使わない**
- reference/ROAD_MAP.md / tasklog / architect メモ / Issue 本文 / コミットメッセージ /
  PR タイトル / ブランチ名のいずれでも **同じ表記** に統一する
- ブランチ名例: `feature/<issue-number>-bolt-<n>-<short-name>`
- コミット例: `feat(bolt-2): implement chroma upsert`

---

## 4. bolt 設計時の出力フォーマット

bolt を設計するときは、必ず以下のフォーマットで出力する。
セクションは省略しない（不要なものは「なし」と書く）。

```markdown
## bolt-n: <bolt名>

### 目的
この bolt で達成したいことを 1〜3 行で書く。

### 作るもの
この bolt で実際に作成・変更する成果物を箇条書きで書く。
ファイル名・関数名レベルまで具体に落とす。

### 作らないもの
この bolt では意図的に扱わないものを箇条書きで書く。
「次の bolt に回す」「別 Issue にする」など、回し先も書く。

### 対象ファイル・ディレクトリ
この bolt で読む・変更する可能性が高いファイルを列挙する。
読むだけのファイルと書くファイルを区別する。

### 実装方針
どのような方針で実装するかを 3〜10 行で書く。
ライブラリ選定や設計上のトレードオフはここに残す。

### ドキュメント更新
README.md / reference/ROAD_MAP.md / reference/ISSUES.md / docs 配下など、
更新すべきドキュメントを書く。なければ「なし」。

### 完了条件
この bolt が完了したと言える条件を箇条書きで書く。
「動作確認」「テスト」「ドキュメント反映」などを含める。

### 確認事項
ユーザーに確認すべき不明点があれば書く。
明確な不明点がない場合は「なし」と書く。

### 次の bolt への引き継ぎ
次の bolt で扱うべきこと、残した課題、注意点を書く。
最後の bolt の場合は「なし（Issue 完了）」と書く。
```

出力場所:

- 通常は **PR 本文の冒頭** か、**作業ログ (`docs/taskLog/`) があればそこ** に書く
- どちらにもない場合は、PR の最初のコメントとして残す

## 5. 「何を作るか」を決める判断基準

bolt の成果物を決めるときは、以下の判断基準に従う。

### 5.1 サイズと責務

- **1 つの bolt では、主目的を 1 つに絞る**
  - 複数の責務を 1 つの bolt にまとめない（レビューと修正の繰り返しが破綻する）
- 1 bolt のコード差分は、目安として **ファイル数 5 前後 / 行数 300〜500 行** までに抑える
- それを超えそうなら、分割し直す

### 5.2 教材としての順序

- **初学者が理解しやすい順番**で並べる
  - 「動く最小構成」→「機能追加」→「品質改善」の順
- **ローカルで動く最小構成を優先**する
  - AWS 移行用の実装は、必要以上に先取りしない
- **RAG 本体は、教材として段階的に育てられる形** にする
  - 親子チャンク・rerank・RAGAS などは、別 bolt として後ろに置く
- **抽象化しすぎない**
  - 学習者がコードを 1 ファイルで追える形を優先する
  - 共通基底クラスを切るのは「2 度以上同じ実装を書く」ことが分かってから

### 5.3 読む / 書くを分ける

- **既存実装を読む bolt と、新しく書く bolt は混ぜすぎない**
  - 読みながら書くと、PR の差分が「何のための変更か」見えにくくなる
- 既存コードを読み解く必要がある場合は、`bolt-N (理解)` のように
  目的を明示する

### 5.4 ドキュメント

- **ドキュメント整備だけの bolt を作ってもよい**
  - README / ROAD_MAP / ISSUES の整合性が崩れたら、まずそれを直す bolt を入れる

### 5.5 設計判断の扱い

- **難しい設計判断が必要な場合は、実装前にユーザーへ確認する**
  - 候補が 2 つ以上あるとき
  - 後戻りコストが大きいとき（スキーマ / 認証 / 課金 / 公開 API / 学習者の動線）
- 確認は AskUserQuestion か、PR の「Review Required」セクションで明示する
- 明らかに合理的に判断できる場合は、前提を **PR 本文に明記** して進める

### 5.6 不可逆な操作

- bolt の中で不可逆な操作（DB 破棄 / 公開ブランチへの force push /
  外部サービスへの本番反映）を含める場合は、必ず **その bolt 専用に隔離** する
- 他の機能変更と混ぜない

## 6. reference/ROAD_MAP.md との関係

- **詳細は reference/ROAD_MAP.md に書かない**
  - reference/ROAD_MAP.md は Phase 単位（学習の地図）
  - bolt の詳細は **PR 本文 or `docs/taskLog/`** に残す
- reference/ROAD_MAP.md には、以下だけ書く
  - このプロジェクトでは bolt 単位で作業を進めること
  - bolt 名は `bolt-0`, `bolt-1`, `bolt-2` の形式にすること
  - bolt の設計には `.claude/skills/bolt-planning/SKILL.md` を参照すること
  - bolt ごとの成果物・完了条件・引き継ぎを明確にしてから実装すること

## 7. tasklog / architect との関係

この skill のスコープは **bolt の設計** であり、tasklog や architect メモの
運用そのものは扱わない。位置づけだけ整理しておく:

- bolt で何を作るかを整理した結果は、必要に応じて `docs/taskLog/` に残す
taskLog の命名規則

taskLog を作成する場合は、以下の命名規則を使用する。

保存先

docs/taskLog/

ファイル名

phase<phase番号>-bolt-<番号>.md

例

phase2-1-bolt-0.md
phase2-1-bolt-1.md
phase2-2-bolt-0.md
phase3-1-bolt-0.md

原則

* 原則として 1 bolt = 1 taskLog
* bolt 完了時に作成する
* 次の bolt への引き継ぎを記載する
* 学習内容・設計判断・実装内容を残す
* 後から学習履歴を追える状態にする

- 設計判断が発生した場合は、必要に応じて `docs/architect/` などに残す
- tasklog / architect の詳細な運用設計は **別 skill or 別 Issue** で扱う

つまり、この skill では tasklog 管理を作り込まない。
**bolt ごとの成果物整理が主目的**である。

## 8. 実施フロー (チートシート)

新しい Issue / Phase に着手するとき:

1. Issue 本文を読み、主目的を 1 行で要約する
2. 目的を満たすために必要な変更を、**§5 の判断基準** に従って bolt に分割する
3. 各 bolt について、**§4 のフォーマット** で設計を書く
4. 設計時点で **確認事項** がある場合は、AskUserQuestion で確認する
5. 1 bolt = 1 PR で実装する
6. PR 本文の冒頭に、当該 bolt の §4 フォーマットを貼る
7. 完了したら「次の bolt への引き継ぎ」を読み、次の bolt へ進む

bolt を実装している途中で、当初の見積もりより大きいと気づいたら:

- いったん作業を止めて bolt を **分割し直す**
- 既に書いたコードがあれば、分割後の bolt のどれに属すか判定する
- 当初の bolt の「作るもの / 作らないもの」を更新して PR に反映する
